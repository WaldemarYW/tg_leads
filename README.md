# tg_leads

Подробное описание текущего автоответчика, его точек входа, сценариев, напоминаний и вспомогательных сервисов.

Проект состоит не из одного скрипта, а из нескольких связанных частей:

- `auto_reply.py` — главный рантайм. Слушает Telegram, ведет кандидата по воронке, пишет в Google Sheets, управляет напоминаниями, FAQ и состоянием.
- `bot.py` — Telegram-бот для оператора. Через него можно запускать и останавливать `auto_reply.py`, смотреть статус и работать с несколькими аккаунтами.
- `mini-sider/server.js` — локальный HTTP-сервис для AI-подсказок и AI-классификации.
- `tg_to_sheets.py` — старый совместимый модуль для ручного экспорта и legacy-логики по Google Sheets.

Если описать систему совсем коротко, то поток такой:

1. Лид приходит из группы или пишет в личку.
2. `auto_reply.py` определяет, можно ли запускать автоворонку.
3. Кандидат переводится в V2-сценарий.
4. Все события складываются в очередь `SheetsQueueStore` и потом долетают в Google Sheets.
5. Фоновые циклы следят за тишиной в диалоге и отправляют напоминания на нужном этапе.

## С чего начинать

Если вы впервые заходите в проект и хотите понять, что реально запускает систему, начинайте в таком порядке:

1. `mini-sider/server.js`
2. `auto_reply.py`
3. `bot.py`
4. Потом уже смотрите вспомогательные модули

Почему именно так:

- `mini-sider/server.js` отвечает за HTTP-эндпоинты, которые `auto_reply.py` использует для AI-ответов, AI-intent и выбора формата.
- `auto_reply.py` — основной рабочий процесс. Именно здесь живет вся бизнес-логика.
- `bot.py` не ведет диалоги с кандидатами, а управляет запуском/остановкой автоответчика и статусами аккаунтов.
- Остальные файлы в основном обслуживают основной рантайм.

## Быстрый запуск

Важно: Python-скрипты читают `.env` не из корня репозитория, а из фиксированного пути `/opt/tg_leads/.env`.

Минимальная схема запуска вручную:

```bash
cd mini-sider
npm install
npm start
```

## Что делает каждый ключевой файл

### Основной рантайм

- `auto_reply.py` — главный процесс. Поднимает Telethon-клиент, ловит входящие и исходящие сообщения, пишет в Sheets, ведет состояние сценария, запускает follow-up loop и flush loop очереди.
- `flow_engine.py` — компактная декларация V2-переходов между крупными этапами.
- `v2_state.py` — JSON-хранилища для V2 enrollment и runtime-state.
- `auto_reply_state.py` — JSON-хранилища follow-up, step-state и локального pause-state.
- `sheets_queue.py` — SQLite-очередь на запись в Google Sheets с retry/backoff.

### Классификация и AI

- `auto_reply_classifiers.py` — локальные правила для вопроса, стопа, подтверждения, выбора формата и запроса "лучше текстом".
- `intent_router.py` — тонкая прослойка над локальным intent-классификатором.
- `faq_service.py` — собирает FAQ-корпус из `faq-for-ai.txt` и `telegraph-faq.txt`, готовит ответы на вопросы кандидата.
- `faq_learning.py` — формирует записи для FAQ-логов.
- `mini-sider/server.js` — HTTP API к OpenAI для генерации HR-ответов и простых классификаций.

### Контент и Sheets

- `content_dispatcher.py` — пересылает кандидату контент по ссылке на сообщение Telegram.
- `tg_to_sheets.py` — общие функции работы с Google Sheets, статусы, локи и legacy-шаблоны.
- `candidate_notes.py` — дописывает ответы кандидата в поле заметок.

### Лиды и регистрации

- `registration_ingest.py` — парсит сообщения из трафик-группы с анкетами/документами.
- `faq-for-ai.txt` — основной FAQ-контент для AI-ответов.
- `telegraph-faq.txt` — дополнительный FAQ-контент.

### Тесты

- `tests/test_flow_engine_v2.py` — переходы V2-flow.
- `tests/test_auto_reply_flow.py` — legacy flow-chain.
- `tests/test_registration_ingest.py` — разбор регистрационных сообщений.
- `tests/test_faq_service_v2.py` — FAQ-нормализация и voice-recap.
- Остальные тесты покрывают state-store, classifiers, очередь и intent-routing.

## Главная логика проекта

### 1. Источники лидов

Система умеет запускать автоворонку из нескольких источников:

- сообщение в группе лидов `LEADS_GROUP_TITLE`
- первое входящее сообщение кандидата в личке
- специальный старт сообщением `"+"`
- тестовый старт для тестового пользователя через `старт8` / `start8`
- отложенный старт на альтернативном аккаунте

При этом проект умеет:

- фильтровать лидов без ПК/ноутбука
- фильтровать лидов по полю `HR` и пересылать их в отдельные группы без запуска автоответчика
- не трогать уже paused-диалоги
- различать primary и alt account
- сохранять владельца лида между аккаунтами

### 2. Основной рабочий режим сейчас: V2

`FLOW_V2_ENABLED = True`, поэтому фактически в проекте главным является V2-сценарий. Legacy-цепочка все еще лежит в коде и работает как совместимость, но основной диалог ведется через V2.

### 3. Запись в таблицы не идет напрямую на каждое действие

`auto_reply.py` старается писать через `SheetsQueueStore`:

- событие кладется в SQLite
- фоновый цикл `sheet_flush_loop()` читает пачками
- при ошибках делает retry с backoff
- при некоторых сбоях есть fallback на прямую запись

Это уменьшает риск потери диалога из-за временных проблем с Google API.

## Полный V2-сценарий

Ниже описан фактический путь кандидата по коду.

### Этап 1. Screening

Скрипт отправляет:

- `SCREENING_INTRO_TEXT`
- `SCREENING_Q1_TEXT` — был ли опыт в дейтинге
- `SCREENING_Q2_TEXT` — возраст

Что происходит дальше:

- если возраст `< 18` — отправляется `AGE_UNDER18_TEXT`, диалог останавливается
- если возраст `> 40` — отправляется `AGE_OVER40_TEXT`, диалог останавливается
- после age reject дополнительно может уйти `REFERRAL_AFTER_REJECT_TEXT`
- если ответы валидны, кандидат переводится на `STEP_COMPANY_INTRO`

Дополнительно:

- ответы кандидата пишутся в заметки
- состояние сохраняется в `V2RuntimeStore`
- для screening действует отдельный более быстрый debounce

### Этап 2. Company intro

Отправляется `COMPANY_INTRO_TEXT`.

Смысл этапа:

- кратко представить компанию
- объяснить, что работа удаленная, 8 часов, только ПК/ноутбук
- предложить голосовое сообщение с деталями

Варианты развития:

- кандидат согласен продолжать и есть `VOICE_MESSAGE_LINK` — пересылается голосовое, шаг меняется на `STEP_VOICE_WAIT`
- голосового нет или кандидат не хочет голосовое — система сразу переводит в блок графика
- если кандидат просит не голосовое, а текст, срабатывает `should_replace_voice_with_text()` и условия пересказываются текстом

### Этап 3. Voice wait

Если голосовое отправлено, система ждет реакцию кандидата.

Варианты:

- кандидат подтверждает, что можно дальше — переход на график
- кандидат пишет, что не может/не хочет слушать — система продолжает текстом
- кандидат задает вопрос — срабатывает FAQ/AI-ответ, после чего шаг не теряется

### Этап 4. Выбор смены

Отправляется `SCHEDULE_SHIFT_TEXT`.

Система ожидает:

- `денна`
- `нічна`
- время в формате, похожем на смену
- явное согласие/несогласие по графику

Ветвления:

- если кандидат спорит с форматом смен, отправляется уточнение, подходит ли вообще фиксированная дневная/ночная смена
- если отвечает "нет" — автоответчик останавливается
- если выбор сделан, система пишет выбор в заметки и идет дальше

### Этап 5. Подробности рабочего процесса

После выбора смены отправляются:

- `SCHEDULE_DETAILS_TEXT`
- `SCHEDULE_CONFIRM_TEXT`

Здесь задача этапа:

- объяснить интенсивность работы
- объяснить командный формат
- подтвердить, что рабочий процесс понятен

Если кандидат не подтверждает продолжение:

- система остается на этом шаге
- может отвечать на вопросы по FAQ/AI
- затем продолжает ждать подтверждение

### Этап 6. Балансы и материалы

Если кандидат подтвердил рабочий процесс:

- через `dispatch_v2_content()` пересылаются `PHOTO_1_MESSAGE_LINK`
- потом `PHOTO_2_MESSAGE_LINK`
- затем задается `BALANCE_CONFIRM_TEXT`

На этом шаге система:

- объясняет блок баланса и оплаты
- ждет подтверждение или вопросы
- пишет FAQ-вопросы в отдельный лист

### Этап 7. Тестовое задание

Если кандидат подтвердил блок баланса:

- пересылается `TEST_TASK_MESSAGE_LINK`
- шаг меняется на `STEP_TEST_REVIEW`

На этом этапе система:

- ждет подтверждение готовности продолжать
- умеет отвечать на вопросы по FAQ
- следит за количеством сообщений и временем ожидания

### Этап 8. Анкета

Когда кандидат пишет, что готов продолжать:

- пересылается `FORM_MESSAGE_LINK`, если он задан
- иначе отправляется текст `FORM_TEXT`
- шаг меняется на `STEP_FORM_FORWARD`

Дальше система ожидает:

- текст анкеты
- фото или скрин документа для верификации возраста

Поведение:

- если пришел текст, похожий на заполненную анкету, скрипт просит фото документа
- если пришло фото, система считает этап завершенным
- затем отправляет сообщение о передаче тімліду и переводит в `STEP_HANDOFF`

### Этап 9. Handoff

После получения анкеты и фото:

- отправляется подтверждение
- дальнейший авто-сценарий фактически завершен
- lead считается переданным человеку

## Что считается вопросом, стопом и продолжением

Ключевые локальные правила лежат в `auto_reply_classifiers.py`.

### Вопрос

Сообщение считается вопросом, если:

- содержит `?`
- содержит ключевые слова вроде `як`, `коли`, `скільки`, `что`, `как`
- выглядит как короткий вопрос даже без `?`

### Стоп

Сообщение считается стопом, если содержит фразы вроде:

- `не актуально`
- `не цікаво`
- `не пишіть`
- `вже знайшов роботу`
- `stop`
- `unsubscribe`

Но короткие ответы вроде `ок`, `ясно`, `питань нема` в стоп не попадают.

### Подтверждение продолжения

Система трактует как `ack_continue` такие ответы, как:

- `так`
- `ок`
- `готова`
- `можемо далі`
- `питань нема`
- `все зрозуміло`

## Как отвечают FAQ и AI

Есть три уровня ответов:

### 1. Локальные правила

Самый быстрый уровень:

- локально определяется intent
- локально определяется выбор формата
- локально определяется stop/continue

### 2. FAQ-ответ

Если кандидат задает вопрос:

- `faq_service.py` собирает общий корпус из `faq-for-ai.txt` и `telegraph-faq.txt`
- AI получает черновик с ограничением отвечать только по FAQ
- ответ должен быть только на украинском и с обращением на `Вы`

### 3. Like training

Если включен `LIKE_TRAINING_ENABLED`, проект умеет обучаться на одобренных ответах оператора:

- кандидатский вопрос и ответ оператора связываются через реакцию
- пара записывается в лист `FAQ_Likes_Train`
- при следующем похожем вопросе система может взять готовый ответ из обучения

Это полезно, когда реальные ответы оператора лучше, чем общий FAQ.

## Напоминания и автодожимы

Это одна из самых важных частей проекта. Напоминания идут не хаотично, а по конкретным таймерам.

### V2 step-wait reminders

Для шагов из `WAIT_STEP_SET`:

- `STEP_SCREENING_WAIT`
- `STEP_COMPANY_INTRO`
- `STEP_VOICE_WAIT`
- `STEP_SCHEDULE_SHIFT_WAIT`
- `STEP_SCHEDULE_CONFIRM`
- `STEP_BALANCE_CONFIRM`
- `STEP_TEST_REVIEW`

Используется трехступенчатая схема:

1. Через `STEP_CLARIFY_DELAY_SEC` отправляется мягкое уточнение
2. Через `STEP_FALLBACK_1_DELAY_SEC` отправляется первый дожим
3. Через `STEP_FALLBACK_2_DELAY_SEC` отправляется второй дожим

Текущие дефолты:

- `STEP_CLARIFY_DELAY_SEC = 600` секунд, то есть 10 минут
- `STEP_FALLBACK_1_DELAY_SEC = 21600` секунд, то есть 6 часов
- `STEP_FALLBACK_2_DELAY_SEC = 259200` секунд, то есть 3 дня

Важно:

- для второго и третьего касания действует глобальный дневной лимит `GLOBAL_FALLBACK_DAILY_LIMIT`
- тексты могут быть переписаны через AI, если включен `AI_FOLLOWUP_REWRITE_ENABLED`
- после каждого реально отправленного напоминания таймер перевооружается от нового момента отправки

### Reminder по фото документа

Если кандидат уже на шаге анкеты, но не прислал фото:

- через `FORM_PHOTO_REMINDER_DELAY_SEC` отправляется напоминание

Дефолт:

- `FORM_PHOTO_REMINDER_DELAY_SEC = 300`, то есть 5 минут тишины

### Legacy follow-up reminders

Если когда-то будет использоваться legacy flow без V2, там есть отдельные follow-up шаблоны:

- через 30 минут
- через 24 часа
- через 3 дня

Они живут в `FOLLOWUP_TEMPLATES` и ограничиваются окном `FOLLOWUP_WINDOW_START_HOUR` / `FOLLOWUP_WINDOW_END_HOUR`.

## Legacy flow: что осталось от старой схемы

Хотя V2 сейчас основной, в проекте все еще есть старая последовательная цепочка через `auto_reply_flow.py`.

Маршрут выглядит так:

1. `CONTACT`
2. `INTEREST`
3. `DATING`
4. `CLARIFY`
5. `SHIFTS`
6. `SHIFT_QUESTION`
7. `FORMAT`
8. `FORMAT_QUESTION`
9. `VIDEO_FOLLOWUP`
10. `TRAINING`
11. `TRAINING_QUESTION`
12. `FORM`

Особенности legacy flow:

- активно использует `send_and_update()`
- может AI-переписывать отдельные сообщения
- умеет отправлять и видео, и миникурс
- хранит шаг в `StepState`
- использует отдельный `FollowupState`

Нужен он в основном для совместимости со старыми диалогами и recovery логики.

## Работа с контентом

Контент для V2 обычно не зашит в текст, а пересылается из Telegram по ссылкам на сообщения:

- `VOICE_MESSAGE_LINK`
- `PHOTO_1_MESSAGE_LINK`
- `PHOTO_2_MESSAGE_LINK`
- `TEST_TASK_MESSAGE_LINK`
- `FORM_MESSAGE_LINK`

Как это работает:

1. `content_dispatcher.parse_message_link()` разбирает ссылку
2. `dispatch_content()` получает исходное сообщение
3. сообщение или медиа пересылается кандидату
4. в таблицу летит preview пересланного контента

Если ссылка отсутствует или сообщение не найдено:

- сценарий не падает сразу
- кандидат получает запасной текст о том, что материал временно недоступен

## State-файлы и локи

Проект активно пишет runtime-state в файлы. Это важно для продакшн-работы и recovery.

### Важные state-файлы

- `AUTO_REPLY_STATUS_PATH` — последний статус отправки
- `AUTO_REPLY_FOLLOWUP_STATE_PATH` — legacy follow-up state
- `AUTO_REPLY_STEP_STATE_PATH` — legacy step-state
- `AUTO_REPLY_PAUSED_STATE_PATH` — paused/active status по peer и username
- `AUTO_REPLY_V2_ENROLLMENT_PATH` — список peer, заведенных в V2
- `AUTO_REPLY_V2_RUNTIME_PATH` — runtime-state каждого V2 peer
- `AUTO_REPLY_SHEETS_QUEUE_PATH` — SQLite-очередь событий на запись в Sheets
- `AUTO_REPLY_FALLBACK_QUOTA_PATH` — дневная квота на fallback-напоминания

### Важные lock-файлы

- `AUTO_REPLY_LOCK` — не дает поднять второй экземпляр автоответчика
- `TELETHON_SESSION_LOCK` — защищает Telegram-сессию
- отдельные lock-файлы используются и для аккаунтов в `bot.py`

## Управляющий бот `bot.py`

`bot.py` нужен не для общения с кандидатами, а для управления процессами.

Он умеет:

- показывать список аккаунтов
- включать и выключать аккаунт
- запускать `auto_reply.py` для выбранного аккаунта
- останавливать его
- читать статус последней отправки
- выгружать чаты за последние `EXPORT_DAYS`

### Multi-account режим

`bot.py` поддерживает несколько аккаунтов через:

- `ACCOUNTS`
- `ACCOUNT_<NAME>_SESSION_FILE`
- `ACCOUNT_<NAME>_AUTO_REPLY_SESSION_FILE`
- `ACCOUNT_<NAME>_STATE_DIR`
- `ACCOUNT_<NAME>_EXPORT_DIR`

Для каждого аккаунта формируются свои:

- lock-файлы
- status-файлы
- state JSON
- queue SQLite

## Регистрации из traffic group

Если задана `TRAFFIC_GROUP_TITLE`, проект следит за сообщениями этой группы.

Что происходит:

1. Ждет сообщение с медиа
2. Берет текст и парсит его через `registration_ingest.py`
3. При наличии `REGISTRATION_DRIVE_FOLDER_ID` загружает файл в Google Drive
4. Пишет результат в лист `Регистрация`

Из текста автоматически пытаются вытащить:

- ФИО
- дату рождения
- телефон
- email
- Telegram кандидата
- график
- дату старта
- город

## HR-фильтр для лидов

В `bot.py` есть раздел `HR фільтр`, где можно завести правила вида:

- `@hr_username -> @target_group`
- `@hr_username -> https://t.me/<group>`
- `@hr_username -> https://t.me/+<invite_hash>`

Как это работает:

1. `auto_reply.py` парсит поле `HR` из сообщения в группе лидов.
2. Если для этого `HR` найдено правило, сообщение один раз пересылается в указанную группу.
3. Ни основной, ни альтернативный аккаунт не запускают onboarding по такому лиду.
4. Лид при этом все равно остается в листе `GroupLeads`.

В `GroupLeads` для этого добавлена отдельная колонка `HR`.

## Mini-sider: что именно поднимает Node-сервис

`mini-sider/server.js` открывает HTTP API по умолчанию на `3000` порту.

Основные эндпоинты:

- `GET /health`
- `POST /dialog_suggest`
- `POST /should_pause`
- `POST /intent_classify`
- `POST /format_choice`

Он использует OpenAI Responses API и переменные:

- `OPENAI_API_KEY`
- `OPENAI_DIALOG_MODEL`
- `PORT`

То есть `auto_reply.py` сам в OpenAI не ходит напрямую. Он общается с локальным Node-сервисом.

## Что важно знать при сопровождении

### 1. Проект уже частично migrated, но не полностью

В коде одновременно живут:

- новый V2-flow
- legacy flow
- legacy таблицы и старые хелперы

Поэтому при правках важно сразу понимать, в какой слой вы лезете.

### 2. Критичная точка правды — `auto_reply.py`

Даже если часть переходов вынесена в `flow_engine.py`, реальная оркестрация все равно живет в `auto_reply.py`.

### 3. Некоторые таймеры и поля выглядят как заготовка под будущую доработку

В проекте есть параметры и структуры, которые сейчас либо используются частично, либо остались как запас под дальнейшее развитие. Перед удалением таких вещей лучше сначала проверить весь runtime-сценарий и тесты.

### 4. Проект рассчитан на recovery после рестартов

Из-за хранения state в JSON/SQLite после рестарта можно восстановить:

- paused/active состояние
- V2 runtime по peer
- ownership лида
- backlog событий в Sheets queue
